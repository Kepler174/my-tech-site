至此，我们完成了《服务器上的 Git》这一章。

相比前面的命令操作，这一章可能显得有点“运维向”。但正如我在开篇所说，理解基础设施对于高级开发者来说是必不可少的。

**回顾一下我们的要点：**

1.  **协议的选择**：我们知道了 `file://`, `http://`, `ssh://`, `git://` 各自的优劣。**SSH** 是私有项目的王者，**Smart HTTP** 是公共互联的基石。
2.  **裸仓库 (Bare)**：我们理解了为什么服务器端必须是 bare repository——为了避免工作目录冲突，保持纯粹的数据存储。
3.  **SSH 密钥链**：我们复习了基于公钥的认证体系，这是 Git 安全模型的地基。
4.  **从小作坊到兵工厂**：
    *   **Level 1**: 用简单的 `authorized_keys` 就能跑起一个多用户 Git 服务。
    *   **Level 2**: 用 `git-shell` 限制用户行为，提升安全性。
    *   **Level 3**: 当需求复杂时，果断转向 **GitLab** 或 **Gitea** 这样的现代化平台，或者直接拥抱 **GitHub**。

在接下来的章节，我们将离开服务器的机房，回到开发者的座位上。
我们将探讨 **分布式 Git (Distributed Git)**。
这听起来有点抽象，但它实际上是在讲“人与人之间如何协作”。在一个没有中央权威的分布式系统里，我们该如何像一个军队一样整齐划一地工作？

那是非常精彩的一章，我们将讨论各种工作流、如何优雅地提交代码、以及如何维护一个大型开源项目。
