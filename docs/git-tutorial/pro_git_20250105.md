这一节本来应该是全书最无聊的，不就是装个软件嘛？但对于 Git 来说，Windows、Mac 和 Linux 上的坑还真不一样。作为过来人，我给点建议。

<!--truncate-->

## Linux 用户

如果你用 Linux，我默认你肯定会用包管理器。

*   **Debian/Ubuntu**: `sudo apt install git-all`
*   **Fedora**: `sudo dnf install git-all`

**经验之谈**：
书里花了大篇幅教你怎么从**源码编译**安装。我的建议是：**别折腾**。
除非你需要开发 Git 本身，或者你的系统里的 Git 版本老得掉渣（比如 CentOS 6 时代的），否则发行版自带的版本完全够用。把时间花在写代码上，别花在解决依赖报错上。

## macOS 用户

Mac 上其实自带了 Git（作为 Xcode Command Line Tools 的一部分）。
你在终端输 `git`，如果没装它会弹窗让你装。

**我的推荐**：
虽然自带的能用，但我强烈建议用 **Homebrew**：
```bash
brew install git
```
为什么？因为系统自带的版本通常更新很慢，而且那是 Apple 修改过的版本。用 Homebrew 能保持你的 Git 是最新鲜的原版，支持新特性。

## Windows 用户 (重点！)

Windows 用户最痛苦。去 [官网下载安装包](https://git-scm.com/download/win) 确实最稳。

但在安装过程中，你会面对**地狱般的“下一步”轰炸**。那个安装向导里可能有十几个配置页！
大部分保持默认就行，但有**一个选项**一定要看清楚：

### 关于换行符 (Line Endings)
安装向导会问你怎么处理 `CRLF` 和 `LF`。
*   **选这个**：`Checkout Windows-style, commit Unix-style line endings` (core.autocrlf = true)
    这就是说：你在 Windows 上写代码（换行是 CRLF），但提交到 Git 仓库时自动转成 Linux 风格（LF）。这能救你的命，特别是当你和用 Mac/Linux 的同事合作时，不然代码全是一行行的 `^M` 差异，会被同事打死的。

### 终端选择
建议选 **Use MinTTY**。那个黑框框其实比 Windows 自带的 CMD 好用点。

---
最后，不管你用啥系统，装完后打开终端敲一下 `git --version`。
看到了版本号？恭喜，我们可以开始搞事情了。
