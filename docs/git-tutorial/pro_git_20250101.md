## 什么是版本控制

版本控制（Version Control）是一种记录一个或若干文件内容变化，以便将来查阅特定版本修订情况的系统。

在没有版本控制系统之前，许多人通过复制文件并重命名（如 `论文_v1.doc`, `论文_v2.doc`）的方式来备份工作。这种做法虽然简单，但极易出错，例如混淆版本或意外覆盖重要文件。

## 版本控制系统的演变

### 1. 本地版本控制系统 (Local VCS)
为了解决手动管理的混乱，人们开发了本地版本控制系统（如 **RCS**）。它的工作原理是在本地磁盘上建立一个数据库，记录文件的所有修订差异（Patch）。通过应用这些差异，系统可以随时重构出文件的任意历史版本。

### 2. 集中式版本控制系统 (Centralized VCS)
为了解决多人协作的问题，出现了集中式版本控制系统（如 **CVS**, **SVN**）。
这也引入了一个单一的集中管理服务器，保存所有文件的修订版本。协同工作的开发者通过客户端连接到这台服务器，取出最新的文件或提交更新。
*   **优点**：便于管理，且能看到其他人在做什么。
*   **缺点**：**拥有单点故障风险**。如果中央服务器发生故障，所有人都无法提交更新；如果磁盘损坏且未备份，所有历史记录将永久丢失。

### 3. 分布式版本控制系统 (Distributed VCS)
Git 属于分布式版本控制系统（DVCS）。
在分布式系统中，客户端不仅仅是检出最新版本的文件快照，而是把代码仓库**完整地镜像**下来，包括完整的历史记录。

这意味着：
*   每一次克隆（Clone）操作，实际上都是一次对代码仓库的完整备份。
*   即使服务器发生故障，也可以利用任何一个客户端的本地镜像来恢复服务器。
*   大多数操作都可以在本地进行（如查看历史、切换分支），无需网络连接，速度极快且高效。

**总结**：
版本控制从简单的本地文件备份，发展到集中式管理，最终演进为分布式的架构，其核心驱动力是对数据安全、协作效率以及去中心化的追求。
