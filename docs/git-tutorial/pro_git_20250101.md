最近重刷了一遍《Pro Git》，也就是大家常说的 Git 圣经。说实话，平时 `git add`、`git commit` 用得挺溜，但真要问我底层是个啥原理，我还真有点虚。
为了不让自己“知其然不知其所以然”，我决定把这次重读的心得整理出来。不搞那些干巴巴的定义，咱们就聊聊我的理解。

## 所谓的“版本控制”

说白了，版本控制就是程序员的**后悔药**。

你写毕业论文或者做设计图的时候肯定经历过这种绝望：
*   `论文_初稿.doc`
*   `论文_修改版.doc`
*   `论文_最终版.doc`
*   `论文_打死不改版.doc`
*   `论文_绝对是最后一版_v2.doc`

这就是最原始的版本控制——**人肉版本控制**。这种方法的缺点显而易见：容易搞混，而且占地方。一不小心覆盖了文件，那就只能哭着重写了。

## 从“个人战”到“团队战”

### 1. 本地时代 (RCS)
为了不让大家手动复制文件，以前有个叫 **RCS** 的东西。它的逻辑很简单：只记录文件每次的**变化部分**（补丁）。想回退版本的时候，就把这些补丁反向打回去。这就好比你只记住了“今天比昨天多吃了一碗饭”，而不是记住了“今天吃了三碗饭”。

### 2. 集中时代 (SVN/CVS)
后来大家要一起干活了，RCS 这种只能在自己电脑上玩的就不行了。这就有了 **SVN**。
SVN 有一个中央服务器，大家都连上去修修补补。
*   **好处**：管理员爽了，谁干了啥一目了然。
*   **坏处**：**服务器是单点故障**。如果服务器断网，大家全歇菜；如果服务器硬盘坏了没备份，那真就是“火葬场”现场，所有历史记录瞬间蒸发，只剩下你手头的一份最新代码。

### 3. 分布式时代 (Git)
这就是我们现在用的 **Git**。
Git 最牛的地方在于：**它不只是下载最新代码，而是把整个代码仓库的“祖宗十八代”全都克隆到你本地**。

这意味着什么？
意味着你电脑里有一份**完整**的备份。哪怕公司的服务器被陨石砸了，随便找个同事的电脑，都能把服务器还原出来。而且因为历史记录都在本地，像查看日志、切换分支这些操作，快得简直不讲道理。

**个人感悟**：
从 RCS 到 Git，其实就是一部**从“怕丢文件”到“怕服务器挂”，最后通过“人人都是服务器”来解决焦虑**的进化史。
