在使用 Git 的过程中，难免会遇到“我是谁？我在哪？这个命令咋用？”的灵魂发问。
技术圈常说 **RTFM** (Read The Fucking Manual)，但对于 Git 来说，直接读手册可能会让你更晕。

<!--truncate-->

## 别轻易用 `git help <verb>`

书上教你用 `git help config` 或者 `man git-config`。
这会打开 Git 的官方手册（Manpage）。

**相信我，除非你想研究语言学或者底层原理，否则别看那个。**
Git 的官方文档写得非常学术、严谨，但也非常枯燥。对于初学者，你看完第一段可能就睡着了。

## `-h` 才是你的真爱

大多数时候，你只是忘了某个参数是 `-a` 还是 `-A`。
这时候，用 `-h`：

```bash
git add -h
```

这会给你一个**极简**的小抄（Cheat Sheet）。没有废话，只有参数列表。这才是我们真正需要的。

## 真遇到问题了怎么办？

如果你遇到了复杂的合并冲突，或者 `detached HEAD` 这种鬼东西：
1.  **Google 报错信息**。
2.  **Stack Overflow**。
3.  **ChatGPT/Copilot**。

现在是 2025 年了，即使是资深程序员，也不需要背诵所有 Git 命令。**懂得如何精准地搜索答案，比背手册重要得多。**

**本章小结**：记住 `git <command> -h` 用来查参数，剩下的交给搜索引擎。

对了，如果你英语还行，Freenode IRC 上的 `#git` 频道依然活跃着一群热心的（也可能是暴躁的）老黑客，那里是最后的求助站。
