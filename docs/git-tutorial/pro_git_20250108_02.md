这是实战篇的开篇。在了解了足够多的理论之后，我们终于要开始在硬盘上留下 Git 的足迹了。
获取一个 Git 仓库（Repository），通常只有两种途径。而这两种途径，对应着两种截然不同的项目启动场景。

<!--truncate-->

## 场景一：从零开始 (git init)

这种场景通常发生在你有一个现有的本地项目，或者你刚刚新建了一个空文件夹准备写代码的时候。你想把这个项目纳入 Git 的版本控制之下。

操作非常简单，进入项目目录，然后输入：

```bash
git init
```

当你按下回车的那一刻，虽然表面上似乎什么都没发生，但在幕后，Git 已经默默地在你的目录下创建了一个名为 `.git` 的隐藏子目录。

**深度解析**：
这个 `.git` 目录就是 Git 的“大脑”。你所有的提交历史、配置信息、分支指针，全部都存储在这个小小的文件夹里。如果你哪天想让这个项目脱离 Git 的控制，只需要删掉这个文件夹，你的项目就会变回一堆普通的纯文本文件。所有的历史记录也会随之烟消云散。

## 场景二：站在巨人的肩膀上 (git clone)

这是更常见的场景：你想参与一个开源项目，或者加入公司现有的开发团队。你需要把远程服务器上的代码拉取到本地。

这时候我们需要使用 `git clone`：

```bash
git clone https://github.com/libgit2/libgit2
```

**Clone 与 Checkout 的本质区别**：
如果你用过 SVN，你可能会习惯说“Checkout 代码”。但在 Git 中，我们用的是“Clone（克隆）”。
这两个词的区别至关重要：
*   **Checkout** 只是检出最新版本的文件。
*   **Clone** 则是**复制整个仓库**。

当你执行 `git clone` 时，Git 会把远程仓库里**每一个文件的每一个历史版本**都拉取到你的 `.git` 目录中。这意味着，即使远程服务器下一秒就炸了，你的电脑里依然保存着这个项目完整的生命史。

## 协议的选择

细心的你可能发现了，`git clone` 支持多种协议（`https://`, `git://`, `ssh://`）。
虽然初学者通常使用 HTTPS 协议（因为它最简单，不用配置密钥），但在职业开发中，配置 SSH 密钥并使用 SSH 协议通常是更推荐的做法，因为它更安全，且能省去每次推送都需要输入密码的繁琐。
